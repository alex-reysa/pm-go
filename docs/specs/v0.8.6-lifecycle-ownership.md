# pm-go lifecycle ownership: ps / stop / recover commands and a port pre-flight

## Objective

Today, when a pm-go run gets tangled — a wedged Docker daemon, a port collision with
a target repo's local stack, a phase-audit workflow ID collision after a supervisor
crash, or a task that committed `node_modules` — the operator has to reach for raw
`docker`, `tctl`, `lsof`, `pkill`, and `git` to recover. The 2026-04-27 Plan B
post-mortem lists this as the most important remaining product gap: pm-go does not
own its own process and workflow lifecycle. This spec closes that gap for the
common cases by introducing a small set of operator-facing commands (`pm-go ps`,
`pm-go stop`, `pm-go recover`), a tracked-PID state file written by the supervisor,
a pre-flight port-conflict check before `pm-go run`/`implement` boots Docker, and
an implementer-side guard that refuses to commit gitignored artifacts. The end
state is that the normal recovery transcript no longer contains `pkill`, `tctl`,
or manual branch surgery.

## Scope

In scope:

- A pm-go-managed PID/state file written by the supervisor when it spawns the
  worker, API, and (when in `implement` mode) the drive child. Format: a single
  JSON file at `~/.pm-go/state/instance-<apiPort>.json` containing `{ apiPort,
  databaseUrl, temporalAddress, startedAt, monorepoRoot, repoRoot, children:
  [{ label, pid, command, startedAt }] }`. Removed on clean supervisor exit.
- `pm-go ps` — read the state file(s), verify each PID is still alive (kill -0),
  print a human table with `LABEL`, `PID`, `PORT`, `UPTIME`, `INSTANCE`. `--json`
  emits the structured form. Stale entries (PID gone) are pruned and reported
  separately.
- `pm-go stop` — graceful shutdown of pm-go-managed processes only. Reads the
  state file, sends SIGTERM to each tracked PID, waits up to a configurable grace
  period (default 5s), then SIGKILL. Replaces `pkill -f pm-go`, which kills
  unrelated monitors. `--instance <apiPort>` targets one instance; default stops
  all.
- `pm-go recover --plan <id>` — inspects API + Temporal state for a plan and
  takes the smallest safe action: if the API is down, restart it via the
  supervisor path; if a phase-audit or completion-audit workflow already exists
  and is still running, attach and wait instead of colliding; if the workflow
  completed but the DB hasn't projected yet, re-run the activity that closes the
  loop; if nothing is salvageable, print the manual remediation hint with
  correctly shell-quoted paths. `--dry-run` prints the plan without acting.
- Port-conflict pre-flight in `pm-go run` and `pm-go implement`: before
  `docker compose up`, check whether `5432`, `7233`, `8233`, and the configured
  API port are bound by non-pm-go processes (i.e. PIDs not in the pm-go state
  file). On conflict, print the offending PID + process command + suggested
  remediation (typically "this looks like the target repo's local stack — stop
  it or pass `--api-port 13001`"), and exit non-zero before touching Docker.
- File-hygiene guard in the implementer's `commitAgentWork` activity: before
  staging files, run `git check-ignore` (or equivalent) over the diff and reject
  the commit if any path is gitignored (catching the `node_modules` symlink
  failure mode from the post-mortem). Mark the task `blocked` with a typed
  `IGNORED_ARTIFACT_COMMITTED` reason and the offending paths in the agent run
  record, so a future `pm-go task repair` can act on it without DB surgery.
- Shell-quoting fix for any human-readable manual recovery hint that pm-go
  prints: every path emitted is wrapped in single quotes (or shell-escaped via
  Node's equivalent of `shell-quote`), so paths like
  `<your-checkout>/999. PROJECTS/pm-go` (a checkout under a directory whose
  name contains spaces) survive copy/paste. A unit test guards a fixture path
  with spaces.

Out of scope:

- Changing pm-go's default host ports (the post-mortem suggests `15432`/`17233`/
  `18233`/`13001`, but moving the defaults is a separate, larger change with
  migration impact). The pre-flight here detects collisions on the existing
  defaults; remapping is left for a future spec.
- Auditable `pm-go task override` / `pm-go phase override` commands and the
  `policy_decisions` audit trail.
- `pm-go conflicts --repo <path>` cross-repo conflict scanning, `pm-go logs`,
  Docker-daemon-state classification in `pm-go doctor --repair`, per-role budget
  CLI flags, planner size-based caps, and the multi-repo release sequencing UI.
- The completion-audit collision-attach behavior (covered by `pm-go recover`'s
  attach-existing path; deeper drive-loop classification of wait failures —
  "API unavailable" vs "workflow running" vs "workflow completed but DB not
  projected" — is partially covered, but a full taxonomy in `pm-go drive` is
  out of scope).
- Worktree pruning, stale stash cleanup, target-repo dev-server detection.

## Constraints

- Must not change the public argv contract of `pm-go run`, `pm-go implement`, or
  `pm-go drive`. New flags only.
- No new top-level runtime dependency. `lsof` and `kill -0` are acceptable
  (already implicit on macOS + Linux). Anything heavier (e.g. `procps`,
  `ps-list`) must justify itself.
- macOS + Linux parity. No `/proc`-only logic; PID liveness checks must use
  `kill(pid, 0)` semantics, not `/proc/<pid>` reads.
- The state file directory (`~/.pm-go/state/`) is created with `0700`
  permissions. State files contain no secrets — only PIDs, ports, command
  argv (without env), and timestamps. `DATABASE_URL` and similar are referenced
  by env-var-name-only, never by value.
- A `Ctrl+C` on the supervisor must leave a consistent state file: either the
  file is removed (clean exit) or each entry's PID is reported as dead by
  `pm-go ps` (crash). No half-written JSON. Use atomic write-then-rename.
- Two supervisors on the same host are allowed if they bind different
  `--api-port` values. State files are keyed by API port to support this.
- `pm-go ps` and `pm-go stop` must run without Docker, without the API being
  up, and without Temporal — they read the state file and signal PIDs only.
  This makes them safe recovery tools when everything else is wedged.
- All new logic must have unit tests that run via `pnpm test` without docker,
  without a real Temporal, and without spawning real child processes (use the
  same dependency-injection pattern as `apps/cli/src/lib/process-manager.ts`
  and `apps/cli/src/status.ts`).
- The implementer hygiene guard must not break the existing stub-runtime
  e2e fixtures. The stub runtime emits clean diffs; the guard simply has
  nothing to reject.

## Acceptance Criteria

State file + `pm-go ps`:

- A new module `apps/cli/src/lib/instance-state.ts` exports
  `writeInstanceState`, `readInstanceState`, `listInstanceStates`, and
  `removeInstanceState`. Unit tests cover atomic write-then-rename, read
  with missing file, and listing across multiple ports.
- `pm-go run` writes the state file after the worker + API are ready and
  removes it on the `stop`/`shutdown` path of `process-manager.ts`. Test:
  injecting a fake `processManager` proves write happens after `httpReady`
  resolves and removal happens during `stop`.
- `pm-go implement` extends the same state file with the drive child entry
  (label `drive`).
- `pm-go ps` with no arguments lists the worker, api, and (if present)
  drive entries with PID, port, uptime, and instance. Stale entries are
  pruned and printed under a `Stale (no longer running)` section. Exit
  code 0.
- `pm-go ps --json` emits a stable JSON shape covered by a unit test.
- After a `pm-go run` with `apiPort=3001`, `pm-go ps` reports exactly three
  entries. After `Ctrl+C` on the supervisor, `pm-go ps` reports zero
  entries within 5 seconds.

`pm-go stop`:

- `pm-go stop` reads the state file and SIGTERMs every tracked PID, waits
  up to `--grace-ms` (default 5000), then SIGKILLs survivors. Removes the
  state file at the end.
- `pm-go stop --instance 3001` only stops the matching instance.
- Unit test: when all children exit cleanly within the grace period, no
  SIGKILL is issued. When one child hangs, it is SIGKILLed.
- `pm-go stop` with no state files prints `(no pm-go instances)` and exits
  0 (idempotent).

`pm-go recover --plan <id>`:

- `parseRecoverArgv` accepts `--plan <uuid>`, `--dry-run`, and `-h/--help`.
  Rejects missing or malformed UUIDs with a clear error and non-zero exit.
- When the API is down, `pm-go recover` brings the stack up via the same
  supervisor entry point used by `pm-go run` and prints "API restored".
- When the API is up and a phase-audit or completion-audit workflow for
  the plan is `RUNNING` in Temporal, recover prints `attaching to existing
  workflow <id>` and waits for completion via the API/worker (no `tctl`
  shelling). Exit code mirrors the workflow result.
- When the workflow has completed but the DB still shows the phase as
  `auditing`, recover invokes the same projection path the worker uses
  and exits 0 once the phase advances.
- `--dry-run` prints the action plan and exits 0 without making changes.
- A unit test exercises each branch with a mocked API client and a
  mocked Temporal client.

Port pre-flight:

- `apps/cli/src/lib/port-preflight.ts` exports `checkPorts(ports: number[],
  knownPmGoPids: Set<number>): PortConflict[]`. Each conflict carries
  `{ port, pid, command }`. PIDs in `knownPmGoPids` are ignored (so
  re-running while pm-go's own stack is up is allowed).
- `pm-go run` and `pm-go implement` call `checkPorts` before
  `docker compose up`. On conflict, exit 1 with: `port <p> is bound by
  pid <pid> (<command>); stop it or pass --api-port <other> to use a
  different API port`.
- Unit test: feeds a fake `lsof`-equivalent that returns a non-pm-go PID
  on 5432; assertion is the exact remediation string and a non-zero exit.
- Unit test: feeds a fake that reports the pm-go-owned worker on 7233;
  assertion is no conflict reported.

File-hygiene guard:

- `commitAgentWork` calls a new `assertNoIgnoredPaths(stagedPaths,
  worktreePath)` helper that runs `git -C <worktree> check-ignore -v
  --stdin` over the changed paths.
- When ignored paths are present, `commitAgentWork` returns a typed
  failure (no commit), the task transitions to `blocked` with reason
  `IGNORED_ARTIFACT_COMMITTED`, and the offending paths are persisted
  on the agent run record (re-using the existing review-finding shape
  is acceptable).
- Unit test: stub `check-ignore` to return `node_modules/foo`; assert
  no commit, task blocked, paths recorded.
- Unit test: stub `check-ignore` to return empty; assert commit proceeds
  unchanged (back-compat with the existing happy-path test).
- Existing stub-runtime end-to-end test still passes.

Shell-quoting:

- Any code path in `pm-go recover`, `pm-go ps`, or `pm-go stop` that prints
  a manual remediation command containing a filesystem path quotes that
  path. A unit test asserts the printed command for a fixture path
  containing a space (e.g. `/tmp/with space/repo`) round-trips through
  `bash -n` (or equivalent parser) without splitting.

## Repo Hints

- `apps/cli/src/run.ts` — supervisor entry; add the post-`httpReady` state-file
  write and the pre-`docker compose up` port pre-flight here.
- `apps/cli/src/implement.ts` — extend the state file with the `drive` child
  entry; reuse the same port pre-flight helper.
- `apps/cli/src/lib/process-manager.ts` — already owns SIGINT/SIGTERM and
  graceful teardown. Hook the state-file removal into its `stop`/`shutdown`
  paths.
- `apps/cli/src/lib/instance-config.ts` — neighbor for the new
  `instance-state.ts` module; mirror its DI style.
- `apps/cli/src/status.ts` — use as the structural template for `ps.ts`,
  `stop.ts`, and `recover.ts` (DI-friendly, no top-level side effects, exec
  injected, write injected, exit code returned).
- `apps/cli/src/index.ts` — register the three new subcommands.
- `apps/cli/src/__tests__/` — add `instance-state.test.ts`, `ps.test.ts`,
  `stop.test.ts`, `recover.test.ts`, `port-preflight.test.ts`. Follow the
  pattern in `status.test.ts` (if absent, look at `run.test.ts`).
- `apps/worker/src/activities/task-execution.ts` — `commitAgentWork` lives
  here. Add the hygiene guard before the stage step.
- `apps/worker/test/task-execution-workflow.test.ts` — mirror the new unit
  tests for the hygiene guard.
- `packages/contracts/src/execution.ts` — add the
  `IGNORED_ARTIFACT_COMMITTED` block reason and the type for the recorded
  offending paths if the existing review-finding shape doesn't already cover
  it.
- `skills/pm-go-orchestration/SKILL.md` — replace the "Don't `pkill -f
  pm-go`" anti-pattern's `Ctrl+C / tracked PIDs` advice with a pointer to
  `pm-go ps` and `pm-go stop`. Add a one-line entry under "Common Failure
  Modes" for the workflow-id-collision recovery path through `pm-go recover`.

## Open Questions

- Should `pm-go recover` learn `--attach-existing` as an explicit flag, or
  is the default "attach when a running workflow is found" behavior enough?
  The post-mortem suggests the explicit flag (`pm-go drive --plan <id>
  --attach-existing`); leaving it for the planner to confirm.
- Where should the typed `IGNORED_ARTIFACT_COMMITTED` reason live — in the
  existing `BlockReason` enum in `packages/contracts/src/execution.ts`, or
  as a new structured field on the agent run? Planner decides.
- Should the state file include the supervisor's own PID (so `pm-go ps`
  shows the supervisor itself, not just its children)? Useful for
  `pm-go stop` to also kill a hung supervisor; pulls in lifecycle questions
  about who removes the file. Default proposal: yes, label it `supervisor`.
