# pm-go v0.8.8 — autonomy + multi-instance: approval iteration, reviewer escalation, coverage hints, health identity, namespaced stack

## Objective

The 2026-04-28/29 AXON dogfood (see `docs/sessions/29-04.md` and
`~/Desktop/pm-go-overnight.txt`) ran three non-trivial plans
(`62081fc2` hardening, `522a8cca` surface-completeness, `8c3b4463`
post-deploy audit) end-to-end. The substantive code shipped to
production. But the run consumed ~14h of operator wall-clock for
~3h of agent work because pm-go's autonomy story breaks at five
specific seams that were not yet captured in the v0.8.7
lifecycle-completeness spec.

This spec closes the autonomy seams that are independent of the
known completion-auditor crash, and adds the architectural change
needed to run multiple `pm-go` instances on the same host without
container-name, volume, or port collisions (multi-instance track).
After v0.8.8 lands, the next AXON-shape dogfood should drive
`spec → completion-audit gate` with zero operator interventions on
a well-formed plan, and should reach `released` with zero
interventions once the separate completion-auditor crash is fixed
or the v0.8.7 completion-audit override path is used. A second
pm-go can run in parallel on the same machine without hand-edited
port arguments.

## Context — gaps surfaced by the AXON run

Each gap has direct evidence in the session log:

1. **`--approve all` is a misnomer.** The flag approves the
   high-band approval tier but skips per-task integration
   approvals. AXON phase 1 hung 10h on a missed integration
   approval; operator detected via `temporal workflow describe`,
   listed pending approvals manually, approved each row by hand.
2. **Reviewer 40-turn cap loops without progress.** Phase 2's
   5-item P-C task hit the cap at 94 + 97 turns across two
   attempts (~$4 spent, ~30min wall-clock). Operator had to
   `temporal workflow terminate task-review-...-1` and
   `override-review` because attempt 3 was guaranteed to repeat
   the same failure mode.
3. **Reviewer approves partial implementations.** The first
   deploy of plan `8c3b4463` only fixed 1 of 4 P-A items. The
   implementer gated `GET /v1/services` but not `POST` / `DELETE`,
   referenced `created_at` instead of `requested_at`, and missed
   the `service_offers.offer_version_id` column. Reviewer passed
   all of it. Operator caught it via post-deploy smoke probe;
   delegated to a focused fix sub-agent for the remaining 3 items.
4. **`/health` does not identify pm-go.** Twice in the same
   session another tsx server (`automation-consultant/backend`)
   silently displaced pm-go's API on port 3001 and returned
   `{"status":"ok"}` from its own health route. `pm-go status`,
   `pm-go drive`, and `pm-go why` all happily talked to the wrong
   process. Operator switched to `API_PORT=3011 pm-go run --repo
   .` to reclaim a free port.
5. **`runCompletionAuditor` reliably crashes on this machine.**
   Operator cannot reach `released` via the workflow path; has to
   synthesize completion-audit rows by hand. The synthesis itself
   hits FK constraints (`completion_audit_reports → agent_runs`)
   that require inserting both rows in the same transaction. The
   v0.8.7 `override-completion-audit` endpoint will paper over the
   symptom; the underlying auditor crash is a separate bug worth
   reproducing.

Multi-instance track:

6. **Single host can run only one pm-go.** Docker compose hard-codes
   service names (`pm-go-postgres-1`, `pm-go-temporal-1`) and ports
   (`5432`, `7233`, `8233`); the API defaults to `3001`. Running a
   second pm-go in parallel on the same host requires hand-editing
   compose files and per-call `--port` flags. Several CLI commands
   (`status`, `doctor`, `why`) reach into containers by hard-coded
   name (`docker exec pm-go-postgres-1 …`).

## Scope

In scope:

- **`--approve all` iterates until empty.** `pm-go drive --approve all`
  re-issues `GET /approvals?planId=<id>&status=pending` after each
  approval and posts approvals for every pending row, across every
  approval subject (`plan`, `task`) and risk band (`high`,
  `catastrophic`). `all` means explicit operator consent to approve
  every row; `none` and `interactive` preserve the current safer
  behavior. If the bulk endpoint skips a row because it is not yet
  eligible, drive falls back to the row's subject endpoint when
  possible and otherwise keeps polling until the row becomes eligible
  or the workflow reaches a terminal state. Loop exits only when
  `pending=0`.
- **Reviewer cap-twice → auto-defer.** `runReviewer` returns a typed
  non-retryable result when the runner hits the model/turn cap:
  `{ ok: false, reason: "TURN_CAP_REACHED", agentRun }`. The
  activity persists the failed `agent_runs` row before control
  returns to `TaskReviewWorkflow`; Temporal must not blindly retry
  this failure outside workflow logic. When two consecutive reviewer
  attempts for the same review cycle end in `TURN_CAP_REACHED`,
  before scheduling the next attempt the workflow asserts at least
  one of: (a) the persisted finding count diverged from the prior
  successful/partial report, (b) the implementer made any code
  change between attempts (`reviewedHeadSha` changed or a new
  implementer commit landed). If neither is true, the workflow
  stamps a deferred review report, records the divergence check in
  `notes`, transitions task to `ready_to_merge`, and exits clean.
  This requires a schema migration that adds
  `review_outcome = 'deferred_post_merge'` and nullable
  `review_reports.notes`.
- **Post-implement coverage check.** New activity
  `verifyImplementationCoverage` runs after the implementer's diff
  lands but before review starts. It uses `tasks.file_scope` only to
  identify the allowed diff window; file scope remains a permission
  boundary, not proof that every scoped path must change. Add an
  optional planner contract field such as
  `tasks.coverageHints = { mustTouchFiles?: string[],
  mustTouchSymbols?: string[] }`. The planner fills it only when the
  spec explicitly names files, endpoints, columns, symbols, or
  mutating surfaces that must change. The activity reports untouched
  `mustTouchFiles` and absent `mustTouchSymbols` from the diff.
  The coverage report is persisted as a task-scoped artifact
  (`implementation_coverage_report`) related to the implementer
  `agent_run`. `TaskReviewWorkflow` loads the latest report and
  injects its findings into the next review pass as auto-generated
  low-severity items so the reviewer starts from coverage facts.
- **`/health` returns service identity.** `GET /health` body
  becomes `{ status: "ok", service: "pm-go-api", version:
  <package.json>, instance: <name>, port: <bound> }`. `pm-go run`
  / `implement` validate this identity during startup, and `pm-go
  status`, `drive`, `why`, `recover` validate
  `service === "pm-go-api"` before trusting an already-running
  endpoint. On mismatch they emit
  `[pm-go] port <port> is held by another service (got: <body>);
  use --port <other> or stop the conflicting process` and exit 1.
- **Project-namespaced compose stack.** Replace the implicit
  `pm-go` compose project name with `pmgo-${instance}` (default
  `pmgo-default`). Containers become `pmgo-${instance}-postgres-1`,
  `pmgo-${instance}-temporal-1`. Volumes become
  `pmgo-${instance}_postgres` by using a generated/per-instance
  compose file whose top-level volume key is `postgres`. CLI
  commands that reach in by name resolve from the active instance's
  state file, not from hard-coded container names.
- **Dynamic port allocation.** New helper `allocateFreePorts(roles[])`
  accepts named roles (`api`, `postgres`, `temporal`,
  `temporalUi`) with preferred ports (`3001`, `5432`, `7233`,
  `8233`) and walks by `+10` increments when a preferred port is
  bound. Allocation is guarded by a per-user lock
  (`~/.pm-go/locks/ports.lock`, or equivalent injected lock in
  tests) so two simultaneous boots cannot pick the same free port.
  The chosen ports are written to the instance config before any
  container starts; the runtime state is updated with PIDs and
  container names after health succeeds. Compose templating reads
  from the resolved config at boot. CLI commands resolve active
  ports from the instance state/config, not from defaults.
- **`--instance <name>` flag everywhere.** `pm-go run`,
  `implement`, `drive`, `stop`, `ps`, `recover`, `status`, `doctor`,
  `why` accept `--instance <name>`; default `default`. Without the
  flag, behaviour is identical to v0.8.7.

Out of scope:

- **Cross-machine instances.** v0.8.8 stays single-host. Remote
  workers / federated state-DBs are a v0.9+ conversation.
- **Reviewer turn-cap raise.** The cap exists for cost control;
  v0.8.8 escalates *behaviour at cap*, not the cap itself. Raising
  to 80 turns is cheap but kicks the same can if the underlying
  task is too large.
- **`runCompletionAuditor` repro.** Tracked separately as
  `docs/specs/v0.8.8-completion-auditor-repro.md` (to file).
- **Auto-shrinking oversized tasks.** When the planner produces a
  task with >5 acceptance criteria, the right answer is "split it
  during planning," not "let the reviewer reject it." Planner
  hardening is a v0.9 spec.
- **TUI updates.** New flags are CLI-driven; TUI shadowing follows
  in v0.8.9.

## Constraints

- Argv contracts for existing `pm-go *` subcommands must stay
  source-compatible. New flags are additive; without `--instance`,
  default behaviour matches v0.8.7.
- `--approve` keeps the current public modes (`all`, `none`,
  `interactive`). Do not add `high-band` / `per-task` as new public
  modes in this slice unless they already exist in a downstream
  deployment; if aliases are needed for compatibility, map them onto
  the row-filtering implementation internally and document them as
  aliases only.
- Instance names must be valid Docker Compose project suffixes and
  safe path segments: lowercase ASCII, `[a-z0-9][a-z0-9-]{0,62}`.
  Reject names containing slashes, dots, uppercase, underscores, or
  shell metacharacters before any file or Docker command is built.
- Canonical instance config/state layout is
  `~/.pm-go/instances/<name>/config.json` and
  `~/.pm-go/instances/<name>/state.json`. Existing helper modules
  that still key runtime state by API port, or inline the default
  state file in `index.ts`, must be reconciled to this layout. The
  config stores desired/resolved ports and compose metadata; the
  state stores live PIDs/container names.
- CLI precedence is: explicit `--api-url` (where supported) wins over
  explicit `--port`; explicit `--port` wins over the active
  `--instance` config; active `--instance` config wins over `.env`;
  `.env` wins over built-in defaults. `--instance` and `--api-url`
  may be combined for read-only diagnostics, but commands should
  warn that the URL bypasses instance port resolution.
- New compose project name must not break existing volumes. Add a
  one-time `migrate-compose` helper that renames the
  `pm-go_pmgo_postgres` volume to `pmgo-default_postgres` (or
  documents the manual step), so existing v0.8.7 installs keep
  their plan history.
- `pm-go stop --instance <name>` stops the local supervisor/API/worker
  PIDs for that instance and runs `docker compose -p pmgo-<name> down`
  for the instance stack, but it must not pass `-v` unless the user
  explicitly asks for destructive volume removal in a future command.
- `/health` body change must be backward-compatible: keep
  `status: "ok"` field; add `service`, `version`, `instance`, `port`.
- All new code paths must run without docker (use injected fakes
  for git, exec, fetch, port-probe, and the docker CLI shim).
- macOS + Linux parity. No shell-out to `lsof`, `netstat`, or
  `/proc`. The port-probe uses `net.Server.listen({ port:
  preferredPort, host: "127.0.0.1" })` to test explicit ports; `port:
  0` may be used only when callers truly want an OS-assigned
  ephemeral port.
- Skill doc at `skills/pm-go-orchestration/SKILL.md` updated with
  the new flags, the `--approve all` iteration semantics, and the
  multi-instance section.

## Acceptance Criteria

Autonomy track:

- `ac-approve-1` `pm-go drive --approve all` against a plan with
  pending approvals across plan-scoped and task-scoped subjects,
  including high and catastrophic risk bands, approves all of them
  in one drive invocation; second invocation is a no-op
  (`pending=0`).
- `ac-approve-2` `GET /approvals?planId=<id>&status=pending`
  returns only pending rows; existing `GET /approvals?planId=<id>`
  remains backward-compatible and returns every row for the plan.
- `ac-approve-3` Drive logs each approved row as a structured,
  parseable event containing approval id, subject, task id when
  present, risk band, approval source (`all` / `interactive`), and
  granted-by. The API still stamps `approved_by` and `decided_at`
  on the approval row.
- `ac-approve-4` Existing `--approve all`, `--approve none`, and
  `--approve interactive` modes keep their current argv contract.
  `none` pauses on the first pending row; `interactive` prompts per
  row; `all` is the only mode that approves every pending row
  without prompting.
- `ac-review-cap-1` When attempt N and N-1 both end in
  typed `TURN_CAP_REACHED` results with identical finding count and
  zero new implementer commits/head changes, the workflow
  auto-defers and the task reaches `ready_to_merge` without
  operator action.
- `ac-review-cap-2` `review_reports.outcome = 'deferred_post_merge'`
  carries a populated `notes` field naming the prior attempts,
  reviewer agent run ids, and the divergence-check result.
- `ac-review-cap-3` When attempt N's findings diverge from N-1
  (i.e., reviewer is making progress), the workflow proceeds to
  attempt N+1 normally; auto-defer does not fire.
- `ac-review-cap-4` The DB migration adds the enum value
  `deferred_post_merge` and nullable `review_reports.notes`;
  existing rows continue to read without migration-time rewrites.
- `ac-scope-coverage-1` `verifyImplementationCoverage` runs after
  implementer success and before review; produces a coverage report
  listing untouched `coverageHints.mustTouchFiles` and missing
  `coverageHints.mustTouchSymbols`.
- `ac-scope-coverage-2` Coverage findings are added to the next
  review pass as `ReviewFinding` items with `source:
  "auto-coverage"`, severity `low`, and `relatedAgentRunId` pointing
  at the implementer run id.
- `ac-scope-coverage-3` When no `coverageHints` are present, or every
  hinted file/symbol is present in the diff, the activity is a
  no-op (no auto-finding).
- `ac-scope-coverage-4` The planner contract, JSON schema, DB task
  JSON shape, prompt rendering, and fixtures accept optional
  `coverageHints` without requiring it on existing plans.
- `ac-scope-coverage-5` The coverage report is persisted as a
  task-scoped artifact with kind `implementation_coverage_report`;
  `TaskReviewWorkflow` loads the latest report for the task and
  includes its findings in the reviewer prompt/input.
- `ac-health-identity-1` `GET /health` body matches
  `{ status: "ok", service: "pm-go-api", version: string,
  instance: string, port: number }`.
- `ac-health-identity-2` `pm-go run` / `implement` fail startup when
  `/health` returns 2xx from a non-pm-go service; they print the
  structured identity-mismatch error instead of treating the port as
  healthy.
- `ac-health-identity-3` `pm-go status`, `drive`, `why`, and
  `recover` against a port held by a non-pm-go service (mocked:
  returns `{"status":"ok"}` only) print the structured
  identity-mismatch error and exit 1; against a matching pm-go on a
  different `--port`, they succeed.

Multi-instance track:

- `ac-instance-1` `pm-go run --instance staging` boots a stack
  whose containers are named `pmgo-staging-postgres-1` and
  `pmgo-staging-temporal-1`; concurrent `pm-go run` (default
  instance) keeps using `pmgo-default-*` and the two stacks share
  no state.
- `ac-instance-2` `pm-go ps` shows both instances when both are
  up; `pm-go ps --instance staging` shows only staging's processes;
  `pm-go stop --instance staging` leaves the default instance
  untouched and does not delete staging's Postgres volume.
- `ac-instance-3` `pm-go drive --instance staging --plan <id>`
  resolves the API port and DB connection from
  `~/.pm-go/instances/staging/{config,state}.json`, not from
  built-in defaults or `.env`.
- `ac-instance-4` Each instance writes Postgres data to its own
  named volume; `docker compose down -v` for one instance does not
  delete the other.
- `ac-instance-5` Invalid instance names such as `../x`, `.x`,
  `My_Stack`, `foo/bar`, and `foo;rm` are rejected before any file,
  process, or Docker command is invoked.
- `ac-instance-6` `pm-go stop --instance staging` runs the
  instance-scoped Docker shutdown (`docker compose -p pmgo-staging
  down`) after stopping tracked local PIDs; it does not pass `-v`.
- `ac-port-alloc-1` When any preferred role port is bound by another
  process, `allocateFreePorts` returns the next free port for that
  role using the `+10` step. The chosen role-port map is written to
  instance config before any container starts.
- `ac-port-alloc-2` Two simultaneous allocations for different
  instances cannot receive the same port for the same role; the
  injected lock serializes allocation in tests.
- `ac-port-alloc-3` Port-probe uses `net.Server` explicit-port
  binding; no shell-out to `lsof`, `netstat`, or `/proc`.
- `ac-port-alloc-4` Generated compose/env wiring maps the resolved
  host ports into Postgres, Temporal, Temporal UI, API child env,
  `DATABASE_URL`, and `TEMPORAL_ADDRESS`.

Cross-cutting:

- All changes covered by unit + integration tests with no real
  docker / postgres / temporal calls.
- `migrate-compose` helper (or documented manual step) leaves
  existing v0.8.7 installs functional — operator runs it once and
  default-instance plans persist.
- Migration/reconciliation covers both state layouts currently
  present in the repo: inline default state at
  `~/.pm-go/instances/default/state.json` and helper state files
  keyed by API port. After migration, commands read the canonical
  per-name instance layout only.
- Skill doc at `skills/pm-go-orchestration/SKILL.md` updated with
  `--instance`, `--approve all` iteration, and identity-check semantics.
- A separate completion-auditor crash repro spec is filed before
  marking v0.8.8 complete, or this spec's objective explicitly
  remains bounded to `completion-audit gate` rather than `released`.

## Phasing recommendation

Ship as four slices so each can land without waiting on the next:

| Slice | Tracks | Estimated wall-clock |
|-------|--------|----------------------|
| v0.8.8.0 | `ac-health-identity-*` | 0.5-1 day |
| v0.8.8.1 | `ac-approve-*` | 0.5-1 day |
| v0.8.8.2 | `ac-review-cap-*`, `ac-scope-coverage-*`, review/task schema migrations | 1.5-2 days |
| v0.8.8.3 | `ac-instance-*`, `ac-port-alloc-*`, state/compose migration helper | 2-3 days |

Slices 0 and 1 alone eliminate the wrong-service and missed-approval
failure modes. Slice 2 reduces reviewer loop toil and gives reviewers
machine-checkable coverage hints. Slice 3 unlocks parallel runs on
one host.

## Appendix — state DB hygiene (informational, not in scope)

The pm-go Postgres volume `pmgo_postgres` persists across
`docker compose up/down`. After ~38h and three large dogfood plans,
the local instance has accumulated:

- 47 `plans` rows (most defunct)
- 162 `worktree_leases` (some pointing at git worktrees that may
  have been pruned)
- 19 `completion_audit_reports` (some hand-synthesized to bypass
  the broken auditor)
- 404 `agent_runs` (some synthesized as FK parents for the audit
  rows above)

This **does not corrupt** new plans — they get fresh UUIDs and
isolated state. It **does** mean global aggregate queries
(`COUNT(*)`, `WHERE status='blocked'`) see historical garbage.

Recovery options are not part of this spec but worth documenting
for operators:

- **Surgical:** `DELETE FROM plans WHERE id IN (…)` with FK
  CASCADE through phases / plan_tasks / agent_runs / audit
  reports / leases. Preserves plans of interest.
- **Nuke:** `cd <pm-go repo> && docker compose down -v` clears
  Postgres + Temporal volumes; next `pm-go run` boots a fresh DB
  via migrations.

A future spec may add `pm-go gc --before <date>` for routine
cleanup; out of scope for v0.8.8.
