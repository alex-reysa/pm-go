# pm-go v0.8.7 â€” lifecycle completeness: snapshot drift, completion-audit override, phase advancement, ID consistency

## Objective

The v0.8.6 dogfood (see `docs/dogfood/v0.8.6-run.md`) successfully
implemented the spec's substantive code but the formal release flow
required seven manual operator interventions to reach completion, and
ultimately could not produce a `released` plan because pm-go correctly
refuses to compound stacked phase-overrides into a release. This spec
closes the lifecycle gaps that drove that toil so the next non-trivial
dogfood run can complete `spec â†’ released` with zero operator
interventions when the run is well-formed, and with one well-defined
override path when it is not.

## Scope

In scope:

- **Snapshot-drift detection.** At supervisor boot and at each
  workflow boundary, compare `git rev-parse <ref>` for the plan's
  configured ref against `repo_snapshot.head_sha`. When they differ,
  emit a structured warning. Add `pm-go run --strict-base` and
  `pm-go implement --strict-base` flags that hard-fail at boot when a
  drift is detected. Default behaviour stays warn-only so existing
  smokes don't break.
- **Completion-audit override path.** Add
  `POST /plans/:planId/override-completion-audit` with the same shape
  as the existing phase override (required `reason` and `overriddenBy`,
  audit-trail row written to `completion_audit_reports.override_*`,
  state-machine guard that the plan is currently `blocked` with a
  matching audit row). Wire `pm-go drive` to surface the override hint
  the same way it surfaces phase-level overrides today. Migration
  `0018_completion_audit_override.sql` adds the columns.
- **Public phase-advancement API.** Add `POST /phases/:phaseId/advance`
  that promotes a `pending` phase to `executing` when (a) every prior
  phase is `completed` and (b) this phase is `pending`. Replaces the
  `psql UPDATE phases SET status='executing'` that operators reach for
  after `/override-audit`. Drive-loop wires it before `POST /tasks/:id/run`
  when it sees a `pending` next-phase.
- **POST /plans returns the generated plan ID.** Today the endpoint
  returns `spec_document_id` as `planId`, but the SpecToPlanWorkflow
  generates a different UUID internally. Fix: have the workflow's
  `persistPlan` activity stamp the plan ID into a row the API can poll
  for, and return that ID from `POST /plans`. `pm-go implement`'s
  plan-persistence wait then polls the right URL.
- **`PM_GO_INTEGRATION_SKIP_VALIDATE` granularity.** Replace the
  single coarse flag with two:
  `PM_GO_INTEGRATION_SKIP_RECURSIVE_BUILD=1` (already exists) and
  `PM_GO_INTEGRATION_SKIP_TESTS=1`. `pnpm install` always runs so the
  auditor's task-test commands can resolve workspace packages.
- **`plans.created_at` honors DB `now()`.** Stop letting the planner
  output's `createdAt` ISO string overwrite the DB default. Plan
  forensics depend on this column being the actual persistence time.

Out of scope:

- Snapshot pinning (`pm-go run --pin-base <sha>` to force a working
  copy reset). Drift detection alone is enough to address the dogfood
  finding; pinning is a heavier behaviour change.
- Cross-repo plans. v0.8.7 stays single-repo per plan.
- Re-architecting the audit's diff calculation to subtract upstream
  commits automatically. That's a larger conversation about what the
  audit baseline means; for this spec the principle is "audits run
  against the recorded base, drift is surfaced not silently
  rewritten."
- TUI changes. All new endpoints are CLI-driven; TUI integration
  follows in v0.8.8+.

## Constraints

- Must not change the external argv contract for `pm-go run`,
  `implement`, `drive`, `doctor`, `status`, `ps`, `stop`, `recover`.
  New flags are additive.
- Must not require new top-level dependencies.
- Migration `0018` must be backward-compatible: existing
  `completion_audit_reports` rows continue to load. New columns are
  nullable.
- Tests must run without docker (use injected fakes for git, exec,
  fetch).
- macOS + Linux parity. No `/proc`-only logic.
- All new endpoints documented in the orchestration skill at
  `skills/pm-go-orchestration/SKILL.md`.

## Acceptance Criteria

- `ac-drift-1` Snapshot-drift detection emits a warning to stderr and
  to `agent_runs.notes` when `git rev-parse <plan.repo_snapshot.ref>`
  differs from `plan.repo_snapshot.head_sha` at supervisor boot.
- `ac-drift-2` `pm-go run --strict-base` exits 1 with a structured
  error message naming the recorded sha and the current sha.
- `ac-drift-3` Drift detection is unit-tested with at least: (a) no
  drift, (b) drift with `--strict-base`, (c) drift without
  `--strict-base` (warning only).
- `ac-completion-override-1`
  `POST /plans/:planId/override-completion-audit` returns 200 with the
  override outcome row, or 409 when the plan is not blocked, or 400
  when `reason` is empty/missing.
- `ac-completion-override-2` Override stamps
  `completion_audit_reports.override_reason`, `overridden_by`,
  `overridden_at` (added by migration 0018) and flips the plan status
  to `released`-eligible.
- `ac-completion-override-3` After override,
  `POST /plans/:planId/release` succeeds end-to-end and produces the
  PR-summary artifact.
- `ac-phase-advance-1` `POST /phases/:phaseId/advance` returns 200
  when promoting `pending â†’ executing` is valid, 409 otherwise.
- `ac-phase-advance-2` `pm-go drive` calls `/advance` automatically
  when it observes a `pending` next-phase after the prior phase
  reached `completed`. No manual `psql` should be needed in the v0.8.6
  dogfood replay.
- `ac-planid-1` `POST /plans` response body contains the
  workflow-generated plan UUID, not the spec_document_id, even though
  the workflow ID still uses the spec id for traceability.
- `ac-planid-2` `pm-go implement`'s plan-persistence wait polls
  `GET /plans/:planId` with the new ID and returns 200 within the
  workflow's normal latency.
- `ac-skip-flags-1` `PM_GO_INTEGRATION_SKIP_TESTS=1` skips the task
  testCommands but still runs `pnpm install` and `pnpm -r build`. The
  integration worktree has populated `node_modules`.
- `ac-skip-flags-2` Setting both flags simultaneously is supported
  and produces the documented behaviour (no test execution, no
  recursive build, install still runs).
- `ac-created-at-1` Inserting a new plan row with no `createdAt` from
  the planner output produces a `plans.created_at` within 1 second of
  the activity start time.
- `ac-created-at-2` Existing rows are not migrated; only new rows
  pick up the change. Migration adds no DDL for this fix; it's a
  code change in `persistPlan`.
- All changes are covered by unit + integration tests; no test runs
  spawn real docker / postgres / temporal.
- Skill doc at `skills/pm-go-orchestration/SKILL.md` updated with the
  new endpoints and flags.
