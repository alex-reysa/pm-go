# pm-go v0.8.8.0 — `/health` service identity

## Objective

When another tsx-loaded server squats on port 3001, pm-go's CLI commands silently
talk to it and return nonsense. The 2026-04-28/29 AXON dogfood lost two operator
windows to this: `automation-consultant/backend` answered `{"status":"ok"}` from
its own health route, and `pm-go status`, `pm-go drive`, and `pm-go why` all
treated the response as successful contact with pm-go.

This slice closes that failure mode by giving `/health` an identity payload and
making the CLI verify it before issuing real requests. It is slice 0 of
`docs/specs/v0.8.8-autonomy-and-multi-instance.md` — the highest-impact /
smallest-scope of the four planned slices, intentionally landed first.

After this slice ships:

- `GET /health` returns a body that uniquely identifies this process as pm-go.
- `pm-go run` / `implement` refuse to treat a foreign-service response as a
  healthy startup, and the operator gets a structured error pointing at the
  conflict.
- `pm-go status`, `pm-go drive`, `pm-go why`, and `pm-go recover` refuse to
  proceed against a port held by a non-pm-go service.

The `instance` field is included now to keep the body shape stable across
v0.8.8 slices, even though real multi-instance support lands in slice 3.
For this slice the `instance` value is the static string `"default"` and the
`port` value is whatever the API actually bound to.

## Scope

In scope:

- `GET /health` body becomes
  `{ status: "ok", service: "pm-go-api", version: string, instance: string, port: number }`.
  The existing `status: "ok"` field is preserved verbatim; the new fields are
  additive.
- `version` is read from the API package's `package.json` (`@pm-go/api`).
  No env / runtime override.
- `instance` is the literal string `"default"` for this slice. Slice 3 will
  replace this with the resolved instance name from the active config.
- `port` is the actual bound port reported by the API listener (i.e., what
  `app.listen` resolves to), not a re-read of `process.env.API_PORT` or a
  hard-coded `3001`.
- A new shared helper `assertPmGoApi(response, body)` in
  `apps/cli/src/lib/api-client.ts` (or equivalent existing module) validates
  that `service === "pm-go-api"` and that `version`/`instance`/`port` are
  present and well-typed. On mismatch it throws a typed
  `PmGoIdentityMismatchError` whose message includes:
  - the URL that was probed,
  - the raw response body (truncated to 200 chars),
  - a one-line remediation: `use --port <other> or stop the conflicting process`.
- `pm-go status`, `pm-go drive`, `pm-go why`, and `pm-go recover` call the
  helper before issuing any real plan/approval/why/recover request. On
  identity mismatch they print the structured error and exit 1 without
  retrying.
- `pm-go run` / `pm-go implement` use the same helper during the API
  startup readiness probe (the existing health-poll loop). On identity
  mismatch they fail startup with the same structured error rather than
  treating the foreign response as a healthy boot.
- The CLI's existing 2xx-only health probe is replaced (not duplicated) by
  the identity-aware probe. There must not be two probes that disagree.

Out of scope:

- Real multi-instance container/volume namespacing — slice 3 (`ac-instance-*`).
- Dynamic port allocation — slice 3 (`ac-port-alloc-*`).
- `--instance <name>` argv flag on commands — slice 3.
- `--approve all` iteration — slice 1.
- Reviewer turn-cap auto-defer — slice 2.
- Coverage hints / `verifyImplementationCoverage` — slice 2.
- Skill doc updates that depend on slice 1/3 (`--approve all` semantics, the
  parallel-pm-go block). The skill doc IS updated in this slice for the
  `/health` identity check, but only that section.

## Constraints

- Backward-compatible body change. Any caller that reads only `status` keeps
  working. Do not rename or remove `status`.
- macOS + Linux parity. No shell-out for the port probe, no `lsof`, no
  `/proc`. The startup readiness probe uses `fetch` against the bound URL.
- All new code paths run without docker / postgres / temporal. Tests inject
  fakes for `fetch` and the package-json reader.
- `assertPmGoApi` is pure: it takes a parsed body (and ideally the URL string)
  and either returns or throws. It never makes a network call itself.
- Argv contracts unchanged. No new flags in this slice.
- Error messages are stable, machine-greppable strings starting with
  `[pm-go] port <port> is held by another service`. Operator runbooks
  (and future automation) will match against this prefix.
- Tests cover all three `ac-health-identity-*` cases plus a regression test
  for the matching-pm-go-on-different-port case (`status` against
  `--port 3011` succeeds when 3011 is pm-go).

## Acceptance Criteria

- `ac-health-identity-1` `GET /health` body matches
  `{ status: "ok", service: "pm-go-api", version: string, instance: string, port: number }`.
  `version` is the `@pm-go/api` package version. `instance` is `"default"`.
  `port` is the actual bound port.
- `ac-health-identity-2` `pm-go run` / `pm-go implement` fail startup when
  `/health` returns 2xx from a non-pm-go service (mocked: returns
  `{"status":"ok"}` only). They print a structured identity-mismatch error
  whose first line begins with
  `[pm-go] port <port> is held by another service` and exit non-zero
  instead of treating the port as healthy.
- `ac-health-identity-3` `pm-go status`, `pm-go drive`, `pm-go why`, and
  `pm-go recover` against a port held by a non-pm-go service (mocked: returns
  `{"status":"ok"}` only) print the same structured identity-mismatch error
  and exit 1 without issuing any plan / approval / recover request. Against
  a matching pm-go API on a different port (e.g., `--port 3011`), they
  succeed exactly as they do today.
- `ac-health-identity-4` Unit tests cover the helper for: matching identity
  (returns), missing `service` field (throws), wrong `service` value
  (throws), malformed JSON body (throws), HTTP non-2xx (throws), and
  non-fetch network error (throws). Each test asserts the error type and
  the error-message prefix.

## Repo Hints

- `apps/api/src/app.ts` — current `app.get('/health', …)` lives here
  unless already split out. Extend the response body. If split, the route
  file is `apps/api/src/routes/health.ts`.
- `apps/api/package.json` — read `version` from this file at module load
  (not at request time) so the cost is paid once.
- `apps/cli/src/lib/api-client.ts` — add `assertPmGoApi(body, { url })`
  helper and a `PmGoIdentityMismatchError` class. Export both.
- `apps/cli/src/lib/api-client.ts` callers — `status`, `drive`, `why`,
  `recover` all live under `apps/cli/src/{status,drive,why,recover}.ts` and
  reach the API through this shared client. Wire the helper at the
  earliest request (typically the first GET against the API base URL).
- `apps/cli/src/run.ts` — find the existing health-poll loop used by
  `pm-go run` (and reused by `pm-go implement` via supervisor). Replace
  the "got 2xx, ready" check with "got 2xx AND `assertPmGoApi` passes".
- `apps/cli/src/implement.ts` — confirm it inherits the run-side probe;
  no duplicate health logic.
- `skills/pm-go-orchestration/SKILL.md` — add a short note under the
  diagnostics section explaining the identity error and how an operator
  resolves it (`--port` or stop the squatter). Keep it scoped to this slice.
- Existing tests for the API health route and the CLI startup probe live
  in `apps/api/src/__tests__/` and `apps/cli/src/__tests__/` respectively.
  Extend them rather than creating new test files.

## Verification

- `pnpm typecheck`
- `pnpm test`
- Manual smoke (after the change ships):
  1. Start a one-line static HTTP server on 3001 returning
     `{"status":"ok"}` and run `pm-go status`. Expect exit 1 with the
     identity-mismatch prefix.
  2. Stop the squatter, run `pm-go run` in one terminal, then
     `pm-go status` in another. Expect success.
  3. `curl http://localhost:3001/health | jq` — expect the new body shape.
