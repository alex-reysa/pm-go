## Why the audits blocked

Three classes of finding, all genuine, none defects in the implementation:

1. **Baseline drift / scope violation.** During the plan run we pushed
   `693c30a` (v0.8.5.2 IPv6 tctl fix) and `a981b7c` (v0.8.5.3 doctor
   test-fixture update) to `main`. Each task branched from main *at the
   time it ran* — so it branched from `a981b7c`, not from the plan's
   recorded `repo_snapshot.head_sha = abf9828`. Reviewers correctly
   reviewed each task on top of `a981b7c`. But the auditor diffs
   `abf9828..integration_head`, so my upstream commits appeared as
   "files modified outside any task fileScope." The auditor's check is
   correct. The data it audited against was stale.

2. **Stale review coverage.** Reviewers see each task's tip
   individually; the integration head is an octopus merge of tips. Any
   interaction effect between tasks isn't reviewed. Real concern,
   broader product gap.

3. **Acceptance tests not executed (phase 1+2 only).** I had set
   `PM_GO_INTEGRATION_SKIP_VALIDATE=1` to bypass an earlier stale-`dist`
   issue. That env var skips the *entire* integration-validation block
   — including `pnpm install` — so the integration worktree had no
   `node_modules` and the auditor's test commands failed to even resolve
   workspace packages. The flag is too coarse.

## Bugs caught and fixed inline (already on main)

- `v0.8.5.2` (`693c30a`) — `pm-go status` and `pm-go doctor` were
  running `tctl --ad localhost:7233` from inside the temporal
  container. The temporal server only binds to the container's bridge
  IP (`172.21.0.3:7233`), so the calls were always refused. Fixed by
  using `tctl --ad "$(hostname -i):7233"` via `sh -c`.
- `v0.8.5.3` (`a981b7c`) — three doctor test fixtures keyed on the old
  arg-string and silently broke when v0.8.5.2 changed the args. Caught
  by the integration test of phase 0. Fixed.

## Open follow-ups (v0.8.7+ candidates)

These are the real product gaps this run surfaced. They belong in the
next dogfood seed:

1. **Snapshot-drift detection.** When `git rev-parse main` differs from
   `repo_snapshot.head_sha` at activity boundaries, the supervisor
   should warn (or fail under `--strict-base`). Pushing fixes to main
   while a plan is in flight is a workflow-discipline issue, but
   pm-go can detect it and refuse to silently audit against a stale
   baseline.
2. **Completion-audit override path.** `/plans/:id/override-completion-audit`,
   with the same shape as `/phases/:id/override-audit`: required
   reason, required `overriddenBy`, audit-trail row. Without this,
   stacked overrides cannot compound to release, which is correct in
   spirit but produces dead-end runs in practice.
3. **Public phase-advancement API.** Phase 0 → Phase 1 promotion
   currently happens inside the audit-success workflow path. After
   `/override-audit`, the next phase remains `pending` and there is
   no API to move it. Operators reach for `psql`. Add e.g.
   `POST /phases/:id/advance` (only valid when prior phases are
   `completed` and this one is `pending`).
4. **`POST /plans` returns the wrong `planId`.** The endpoint returns
   `spec_document_id` as `planId`, but the workflow generates a
   different plan UUID internally during persistence. The supervisor's
   `pm-go implement` then polls `GET /plans/<spec-doc-id>` and 404s
   forever. This was the root cause of the original 20-minute timeout.
5. **`plans.created_at` is wonky.** Recently-generated plans show
   `2026-04-27 00:00:00+00` (probably planner-supplied, not DB
   `now()`). Forensically misleading.
6. **`PM_GO_INTEGRATION_SKIP_VALIDATE=1` is too coarse.** It currently
   short-circuits before `pnpm install`. Auditor-side test execution
   needs the install step. Either split into
   `SKIP_VALIDATE_TESTS=1` vs `SKIP_VALIDATE_INSTALL=1`, or default
   the install step to always run and only skip the build / test
   commands.
7. **Stale review coverage.** Reviewers see each task tip but never
   the integration octopus-merge head. A post-integration sanity
   review (or auditor-only mode) on the merged head would catch
   inter-task interaction defects the per-task reviews can't.

## Process discipline (to remember next time)

- **Don't push to `main` while a pm-go plan is in flight on the same
  repo.** Even small fixes change the audit baseline.
- **Before launching `pm-go implement` on a non-trivial spec, ensure
  the spec scope fits one Opus planning call** (the 20-minute
  supervisor wait is the hard ceiling on `pm-go implement`; for
  larger specs use `pm-go run` + manual `POST /plans` + `pm-go drive
  --plan <id>`).
- **`pm-go run` + `pm-go drive` is the safer dogfood pattern than
  `pm-go implement` until the v0.8.7 lifecycle gaps land.**
