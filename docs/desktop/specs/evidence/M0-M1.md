**Bugs discovered but NOT fixed in this run** (filed here for follow-up):

- **`createLease` ignores `phase.baseSnapshotId`** (`packages/worktree-manager/src/create-lease.ts:75`). The git worktree is always cut from `HEAD` of the working repo, never from the snapshot SHA recorded on the phase. This means Phase N+1 tasks only see Phase N's work if (a) Phase N's audit passes the happy path and `fastForwardMainViaUpdateRef` advances the working branch, or (b) someone manually merges the integration branch. Neither happened here because the audit was overridden. The override-audit path needs to ALSO do the fast-forward.
- **`override-audit` does not call `fastForwardMainViaUpdateRef`**. The audit happy path advances main + stamps next phase baseSnapshotId + flips next phase status. The override path only flips current to `completed`. After `283c590` the next-phase status is now advanced, but the fast-forward step is still missing. Without it, subsequent phases still get the pre-merge HEAD.
- **`task-exec-${taskId}` workflow-id has no cycle suffix** (`apps/api/src/routes/tasks.ts:181`), unlike `task-review-${taskId}-${cycle}` and `task-fix-${taskId}-${cycle}`. Re-running a task after a manual reset relies on Temporal's `ALLOW_DUPLICATE_FAILED_ONLY` happening to allow start when the previous workflow is in a non-success state; if previous is `COMPLETED`, a new POST /run silently fails or starts (depending on policy). Should adopt the cycle suffix.
- **`merge_runs.startedAt == completedAt`** (`apps/worker/src/workflows/phase-integration.ts:314`). `startedAt` is captured right before `persistMergeRun` at the END of the integration loop, not at the start. Makes debug logs about integration duration meaningless.
- **`validatePostMergeState` swallows logs**. When the activity returns `{ passed: false }`, the `logs[]` array containing the actual pnpm install/build output is discarded. The merge_run record only has `failed_task_id` with no reason — operator can't debug.
- **Supervisor stdout buffering**. The `pm-go run` log file stopped receiving worker/api output after ~3:50 AM despite both children continuing to run. All Phase-1 implementer/reviewer activity was invisible in the log.
