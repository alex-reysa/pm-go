## Findings (root-cause grouped)

### B1 ŌĆö Default CLI dispatch silently swallows `--spec`

**Symptom:** `pm-go --repo X --spec Y --approve all` boots the stack but step 6 prints `no --spec provided; skipping plan submission` (apps/cli/src/run.ts:665). The supervisor runs without ever planning.

**Cause (verified):** With no subcommand, `resolveCliDispatch` routes to `pm-go run`, but the flag-forwarding for the default path drops `--spec`. The supervisor's `submitSpecAndPlan` guard at run.ts:824 (`if (!opts.specPath) throw ŌĆ”`) never triggers because the absence is detected one level up at run.ts:665.

**Severity:** **HIGH ŌĆö silent footgun.** Operator sees a successful-looking stack boot and assumes pm-go is working. Wastes an entire dogfood iteration plus a manual SQL cleanup.

**Reproduction:** `pm-go --repo . --spec <path> --approve all` ŌĆö banner omits the `spec:` line; step 6 says "no --spec provided".

**Counter-evidence:** `pm-go agent --spec X` and `pm-go implement --spec X` both pass the spec correctly (banner shows `spec: <path>`). The bug is scoped to the no-subcommand default route.

---

### B2 ŌĆö `pm-go agent --spec` exits and tears the stack down before the agent loop runs

**Symptom:** Stack comes up, planner workflow is scheduled in Temporal, then the parent `pm-go agent` process exits within ~60s and tears down API + worker. The agent never engages the operator loop; orphan workflow is stranded.

**Cause (suspected):** The agent CLI path likely treats the supervisor handoff as a one-shot rather than a long-running session. When `submitSpecAndPlan` returns its planId (or hits any error), the parent doesn't enter the agent's interactive loop and the process terminates normally ŌĆö which fires the stack-down handler.

**Evidence gap:** Detailed log was overwritten when I `: > .dogfood-logs/M0-M1.log` for the third attempt. Need to retain logs by appending or per-attempt filenames.

**Severity:** **HIGH** if confirmed ŌĆö this is the v0.9 agent-first activation path. Without it, the v0.9 skill update is unusable.

---

### B3 ŌĆö Plan-persistence wait fires false-positive timeout + misleading recovery message

**Symptom:** `[plan-persistence] still waiting, elapsed Nm 0s` continues past the 20-minute ceiling and then errLogs a recovery hint that points the operator at a non-existent UUID.

**Cause (verified end-to-end):**
- API `POST /plans` (apps/api/src/routes/plans.ts:108) generates `planId = randomUUID()`, starts `SpecToPlanWorkflow` with `workflowId: plan-<specDocumentId>`, returns `202` with `planId`.
- The actual plan row is only written by the `generatePlan` activity inside that workflow (Temporal history step 5 confirms `ActivityTaskScheduled {ActivityType: generatePlan, StartToCloseTimeout: 30m0s, MaximumAttempts: 3, RetryPolicy.InitialInterval: 1s, BackoffCoefficient: 2, MaximumInterval: 30s}`).
- A non-trivial spec (the M0+M1 spec is 186 lines, ~12KB) takes the planner agent many minutes; the activity can legitimately run >20 min.
- The CLI polls `GET /plans/<planId>` with `PLAN_PERSISTENCE_TIMEOUT_MS = 20m` (apps/cli/src/run.ts:880). Times out before the activity completes.
- On timeout, the error message at run.ts:897 advises operators to look for "a plan persisted under a different UUID" via `GET /spec-documents/<id>/plan` or `/plans` ŌĆö but in this failure mode **no plan row exists yet under any UUID**. The hint sends operators chasing a phantom.

**Severity:** **MEDIUM-HIGH.** This is the v0.8.8 documented "planId race" memory in `project_pm_go_known_bugs.md`, but the actual mechanism is different: not a race, it's a timeout-vs-activity-runtime mismatch with a wrong diagnostic message.

**Confirmed live state at write-time:**
- Three `SpecToPlanWorkflow`s open: `plan-f14d6aea-ŌĆ”`, `plan-ae6edfb1-ŌĆ”`, `plan-c66e7d97-ŌĆ”` (started 23:00, 23:04, 23:08).
- Zero plan rows from 2026-05-10 in `plans`.
- Worker is currently dead, so all three workflows are stuck on `generatePlan` retry.

### B4 ŌĆö Stale `agent_runs` from killed supervisors require manual SQL cleanup

**Symptom:** Restarting after a supervisor kill leaves `agent_runs` rows with `status='running'`, `completed_at=NULL`, `plan_id=NULL`. The new boot emits FK errors against the orphaned rows; `pm-go recover` doesn't reset them.

**Cause:** `agent_runs` is populated in fire-and-forget style. When the owning process dies, nothing flips the row to `failed`. The on-startup recovery flow (`pm-go recover`) only touches `pm-go`-state files, not DB rows.

**Severity:** **MEDIUM.** Not a blocker ŌĆö manual `UPDATE agent_runs SET status='failed', completed_at=now()` unblocks ŌĆö but it's surprising and undocumented; first-time operators won't know to do this.

## Fix plan

Sorted by severity ├Ś dogfood-blocking impact. The first three should ship before another v0.9 dogfood attempt.

### F1 ŌĆö Make `pm-go --spec` (no subcommand) error or forward (fixes B1)

- **Option A (safest):** When `resolveCliDispatch` falls through to the default `run` route and `--spec` (or any `--`-flag the supervisor consumes) is present, forward it. Add a unit test in `apps/cli/src/__tests__/cli-dispatch.test.ts` that asserts `pm-go --spec X --repo Y` is equivalent to `pm-go run --spec X --repo Y`.
- **Option B (stricter):** Reject unknown top-level flags before dispatch ŌĆö emit `error: --spec only valid with 'agent', 'run', or 'implement' subcommand; did you mean 'pm-go agent --spec X'?`
- Recommendation: **A**, because that's the obvious operator mental model and matches the existing supervisor banner that already prints a `spec:` line.

### F2 ŌĆö Fix `pm-go agent --spec` lifecycle so the agent loop actually runs (fixes B2)

- Add per-attempt logging (`.dogfood-logs/agent-<ts>.log`) so the next failure has a forensic trail.
- Trace where the `pm-go agent` parent exits before the operator loop starts. Most likely: a missing `await` on the agent session in `apps/cli/src/index.ts` near the `createAgentStackController(options, deps)` call (index.ts:754).
- Add an end-to-end smoke that boots `pm-go agent`, asserts the agent session is open for Ōēź 60s, then sends Ctrl+C.

### F3 ŌĆö Make plan-persistence diagnostics honest + tunable (fixes B3)

- **Diagnostic fix (cheap):** When the 20-min ceiling fires, check Temporal workflow status (`OPEN` / `COMPLETED` / `FAILED`). If still `OPEN`, change the message to `the planner activity is still running (workflow plan-<spec-id>); rerun with --plan-wait 60m to wait longer, or run pm-go status to monitor`. Only mention "different UUID" when the workflow is `COMPLETED` but the plan row is missing ŌĆö which would be the actual planId-mismatch failure mode.
- **Knob:** add `--plan-wait <duration>` to `pm-go run` and `pm-go implement` to override `PLAN_PERSISTENCE_TIMEOUT_MS`.
- **Right-size the default:** bump from 20m to 45m for `implement`, keep 20m for `agent` (where the operator can poll manually). The current 20m was tuned against the small AXON specs in `git log`, not against larger MVP-class specs.

### F4 ŌĆö `pm-go recover` resets stale `agent_runs` (fixes B4)

- Extend the recovery routine to `UPDATE agent_runs SET status='failed', completed_at=now(), error_reason='recover: orphaned by previous supervisor' WHERE status='running' AND completed_at IS NULL`.
- Document in `pm-go doctor` output when there are >0 stuck rows.

### F5 ŌĆö Backfill `plan_id` on orchestrator `agent_runs` (fixes B5)

- Lower priority; only affects audit. Trace the call site that inserts orchestrator agent_runs and thread the `planId` through.

### F6 ŌĆö Quiet the Temporal probe banner (fixes B6)

- One-line message change. Skip the warning if the worker's first heartbeat lands within 10s of `[2/6]`.

## How this changes the M0+M1 plan

The dogfood plan assumed pm-go could ingest the M0+M1 spec and produce a plan within the 20m ceiling. Empirically, the spec is large enough that the planner activity runs longer than the CLI poller's patience. Two paths forward, in priority order:

1. **Patch F3's diagnostic (low risk) + retry**: Change the 20m default to 60m on `implement`, fix the misleading message, rebuild the CLI, re-run. Wall-clock cost: ~30 min to ship F3 patch, then 30ŌĆō60 min for planner to actually finish. **Recommended.**
2. **Bypass the CLI poller**: Start the stack manually (`docker compose up`, `pnpm dev:api`, `pnpm dev:worker`), POST the spec and plan via curl, poll the plan endpoint directly with `until` semantics, then `pm-go drive --plan <id>`. No code changes; surfaces what should be a one-shot CLI flow as a multi-step manual flow. **Not recommended ŌĆö it hides the bug from the next operator.**

The third option ŌĆö splitting the M0+M1 spec into smaller pieces that fit under 20m ŌĆö is a workaround that masks B3 without fixing it, and was already covered by my note in the spec that we'd combine M0+M1 for simplicity. Don't go there.
